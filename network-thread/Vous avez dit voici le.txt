Voici mes réponses pour PARALLAX en particulier (pas des réponses génériques) :
Q1 — Protocole réseau : TCP + UDP (hybride)
Pour PARALLAX spécifiquement, voici pourquoi :
TCP pour les données de calcul (workloads, résultats partiels, code source à exécuter, messages d'inscription, élections). Ces messages sont gros (potentiellement plusieurs MB pour un workload), critiques (perdre un résultat = recalculer toute une sous-tâche), et leur ordre compte. Réimplémenter la fiabilité de TCP au-dessus d'UDP serait du travail gaspillé — c'est exactement ce que TCP fait déjà.
UDP pour les heartbeats et le gossip. Tes specs (document 1, section 2.3.11) prévoient un heartbeat toutes les 2 secondes et du gossip vers k = ⌈log₂N⌉ voisins toutes les 5 secondes. Avec TCP, ça veut dire maintenir N connexions ouvertes en permanence sur chaque nœud — coûteux, surtout sur les Wyse (32-64 MB de RAM). UDP est sans connexion : un sendto(), terminé. Si un heartbeat se perd, le suivant arrive 2 secondes après — la perte est tolérée par design (c'est même le principe : l'absence de heartbeat = panne).
Pourquoi pas UDP+ACK custom partout : tu réécrirais TCP en moins bien. Le kernel Linux a 30+ ans d'optimisation TCP, ne luttons pas contre.
Pourquoi pas TCP partout : les heartbeats sur TCP dégradent la détection de panne. Si TCP retransmet pendant 30 secondes avant d'abandonner, tu détectes la panne 30s trop tard, alors qu'avec UDP tu décides toi-même (Tsuspect = 4s, Tfailed = 8s comme dans tes specs).
Q2 — Message queue locale : POSIX mqueue (mq_open)
Pour ton cas :
Avantages POSIX mqueue : API moderne et propre (mq_send, mq_receive, mq_timedreceive), supporte les priorités natives (utile : un message d'élection passe avant un résultat de calcul), bloquant ou non-bloquant au choix, persistance dans /dev/mqueue (visible avec ls /dev/mqueue/), s'intègre avec select()/poll()/epoll() via mq_getattr() + descripteur (essentiel pour ton Network Thread qui doit surveiller à la fois les sockets ET les queues sortantes simultanément).
Pourquoi pas System V (msgget) : API plus ancienne, plus verbeuse, pas d'intégration avec epoll, pas de notification asynchrone propre. C'est l'héritage Unix des années 80 qu'on évite quand on a le choix.
Pourquoi pas custom shared memory + futex : tu passerais une semaine à debugger des race conditions au lieu de coder PARALLAX. À garder pour plus tard si tu as un vrai bottleneck mesuré (peu probable vu la charge).
Petit gotcha : sur Ubuntu Server 18.04, vérifie que /dev/mqueue est monté (mount | grep mqueue). Si non : sudo mkdir /dev/mqueue && sudo mount -t mqueue none /dev/mqueue. Et la limite par défaut de taille de message est 8192 octets — augmentable via /proc/sys/fs/mqueue/msgsize_max si tu en as besoin.
Q3 — Architecture binaire : Lib commune + 3 binaires
Voici pourquoi c'est le meilleur compromis pour PARALLAX :
Le Network Thread est essentiellement le même partout — poll des queues sortantes, envoi sockets, réception sockets, dispatch vers queues entrantes, retry. La logique de base est identique pour Worker, Controller et Master.
Mais les détails diffèrent :

Le Worker a une queue entrante worker_in, le Controller a controller_in, le Master a master_in
Le Master initie des connexions TCP vers tous les workers (pattern hub)
Le Worker accepte des connexions du Master uniquement (pattern spoke)
Le Controller fait du UDP multicast pour les heartbeats

Les fichiers source (src/)
net_protocol.h et net_protocol.c — Le format des paquets
C'est la couche la plus basse, le "langage" que parlent toutes les machines entre elles. Ce fichier définit à quoi ressemble exactement un paquet PARALLAX qui circule sur le réseau.
Concrètement, il définit :

Le header de 64 octets présent au début de chaque paquet (magic number PLXA, version du protocole, type de paquet, UUID source, UUID destinataire, taille du payload, CRC32, etc.)
La liste des types de paquets possibles (PKT_HELLO, PKT_HEARTBEAT, PKT_TASK_ASSIGN, PKT_TASK_RESULT, etc.)
La liste des services destinataires (SVC_EXECUTION, SVC_MONITORING, SVC_PANNE_DETECTOR, etc.)
Les fonctions de sérialisation : transformer une structure C en suite d'octets prête à être envoyée sur le réseau, et l'inverse
Le calcul du CRC32 pour détecter la corruption des paquets

Sans ce fichier, deux machines ne pourraient pas se comprendre. C'est l'équivalent de la grammaire d'une langue.
net_config.h et net_config.c — La configuration et la table des pairs
Deux responsabilités regroupées dans le même fichier parce qu'elles sont étroitement liées :
Première responsabilité : lire le fichier de config INI au démarrage. Le fichier worker.ini ou controller.ini contient l'UUID du nœud, son rôle, ses ports, ses services locaux, le chemin du journal de retry, les paramètres de heartbeat. Ce fichier transforme tout ça en structure C net_config_t.
Deuxième responsabilité : la table des pairs. Une structure qui stocke en mémoire la liste des autres nœuds connus, avec leur UUID, leur IP, leurs ports et la dernière fois qu'on les a vus. Cette table est vide au démarrage et se remplit au fur et à mesure qu'on reçoit des messages HELLO. Elle est protégée par un verrou de lecture/écriture pour que plusieurs threads puissent y accéder en même temps sans corruption.
Il contient aussi les helpers UUID (conversion hex ↔ binaire, comparaison, etc.).
net_retry.h et net_retry.c — La file de retry persistante
Quand le Network Thread essaie d'envoyer un paquet et que ça échoue (destinataire injoignable, connexion TCP cassée, etc.), il ne jette pas le paquet à la poubelle. Il le sauvegarde dans un journal sur disque (un fichier append-only) et le réessayera plus tard.
Ce fichier gère :

L'écriture sur disque de chaque paquet en échec (avec son nombre de tentatives et la date du prochain essai)
Le rejeu au démarrage : si le Network Thread crash et redémarre, il relit le journal et reprend là où il s'était arrêté
Le backoff exponentiel : un paquet qui échoue est réessayé après 500ms, puis 1s, puis 2s, puis 4s, etc., jusqu'à 60s maximum
L'abandon : après 5 tentatives par défaut, le paquet est définitivement abandonné
La compaction : toutes les 60 secondes, le journal est réécrit en supprimant les paquets livrés ou abandonnés, pour qu'il ne grossisse pas indéfiniment

C'est ce qui distingue un vrai système distribué d'un prototype : la fiabilité face aux pannes.
net_core.h et net_core.c — Le cœur du Network Thread
C'est le fichier le plus gros et le plus important. C'est lui qui assemble tout le reste et qui contient la logique principale.
Il fait six choses en parallèle :

Écoute le réseau (socket TCP en écoute pour les données, socket UDP pour les heartbeats et HELLO) via epoll
Lit les queues POSIX sortantes des services locaux dans un thread dédié, pour récupérer les paquets que les services veulent envoyer
Dispatch les paquets reçus vers la bonne queue POSIX entrante selon le service_id du paquet
Envoie les paquets : choisit TCP ou UDP selon le type, ouvre une connexion TCP si nécessaire, gère les échecs en envoyant le paquet vers le retry
Gère le protocole HELLO : émet un HELLO broadcast au démarrage et toutes les 30 secondes, réagit aux HELLO reçus en les ajoutant à la table des pairs
Rejoue le retry toutes les 500ms, et compacte le journal toutes les 60 secondes

C'est le chef d'orchestre. Tous les autres fichiers sont des outils qu'il utilise.

worker.ini, controller.ini, master.ini — Trois exemples de config
Ces trois fichiers ont exactement la même structure mais des valeurs différentes. C'est ce qui distingue un nœud worker d'un nœud master pour le Network Thread.
Différences principales :

UUID différent par nœud (les aaaa..., bbbb..., cccc... sont juste des exemples, à régénérer en vrai)

Pourquoi vboxnet0 a disparu
vboxnet0 est une interface virtuelle créée par VirtualBox uniquement quand au moins une VM est en cours d'exécution avec une carte host-only attachée. Si tu as éteint tes deux VMs, l'interface disparaît automatiquement.
Tu n'as pas besoin de mettre l'IP statique manuellement — VirtualBox la gère automatiquement quand tu démarres une VM. Mais voici comment t'assurer que tout est bien configuré.
La bonne approche
Option A — Démarrer une VM (le plus simple)

sh# Demarrer le service vboxnet0 sans VM

sudo VBoxManage hostonlyif ipconfig vboxnet0 --ip 192.168.56.1 --netmask 255.255.255.0
Option C — Si l'interface existe mais sans IP

Le retry sélectionne la première entrée prête à être rejouée, dans l'ordre du tableau. Ce n'est pas un FIFO strict basé sur l'ordre d'insertion mais un FIFO basé sur les slots libres. En pratique :

Si tu pushes A puis B et que A échoue puis B échoue, ils sont aux slots 0 et 1.
Quand le retry tente de les rejouer, il essaie le slot 0 (A) avant le slot 1 (B). Donc l'ordre est préservé dans la plupart des cas.
Mais si A est rejoué avec succès puis qu'un nouveau paquet C arrive et prend le slot 0 libéré, C peut être rejoué avant B (qui était au slot 1).

Pour PARALLAX en pratique c'est acceptable : les paquets en retry sont par définition des paquets dont l'envoi a échoué (destinataire injoignable). Quand on les rejoue, on essaie juste de les livrer "best effort". L'ordre strict n'est pas critique.

Le problème que résout Makefile
Imagine que tu as un projet avec plusieurs fichiers .c (comme ton Network Thread avec 5 fichiers). Pour compiler à la main, tu devrais taper :

6 commandes. À chaque modification. C'est pénible.
Avec un Makefile, tu tapes juste :

Et il fait tout pour toi. Make est un automatiseur de commandes de compilation.
Mais ce n'est pas juste un raccourci
L'intérêt vraiment important de Makefile, c'est la compilation incrémentale. Make est intelligent : il regarde quels fichiers ont été modifiés depuis la dernière compilation, et recompile uniquement ceux-là.
Exemple concret avec ton projet :
Premier make : tous les fichiers sont compilés (les 5 .c).
gcc -c nt_common.c -o nt_common.o
gcc -c network_thread_start.c -o network_thread_start.o
gcc -c network_thread_hello.c -o network_thread_hello.o
gcc -c network_thread_shutdown.c -o network_thread_shutdown.o
gcc -c main.c -o main.o
gcc *.o -o parallax_netthread -lrt -lpthread
Tu modifies ensuite seulement network_thread_hello.c. Deuxième make :
gcc -c network_thread_hello.c -o network_thread_hello.o
gcc *.o -o parallax_netthread -lrt -lpthread
Make a vu que nt_common.c, network_thread_start.c, network_thread_shutdown.c et main.c n'avaient pas changé → il n'a pas refait leur compilation, il a juste recompilé network_thread_hello.c et refait l'édition de liens.
Sur un petit projet comme le tien, on gagne quelques secondes. Sur un gros projet (genre le noyau Linux qui a 30 millions de lignes), ça peut éviter 2 heures de recompilation.
Comment Make sait quoi recompiler
Make regarde les dates de modification des fichiers (le timestamp mtime du système de fichiers).
Pour ton Makefile, on a écrit :
makefilenetwork_thread_hello.o: network_thread_hello.c network_thread_hello.h nt_common.h nt_types.h
Ça veut dire : « le fichier network_thread_hello.o dépend de ces 4 fichiers ». Si n'importe lequel de ces 4 fichiers a une date plus récente que network_thread_hello.o, alors Make recompile.
Donc si tu modifies juste nt_types.h (ajout d'une constante), Make détectera que ce changement impacte tous les fichiers qui incluent nt_types.h et recompilera tout ce qui en a besoin.

Le problème que résout net_state_t
Imagine que tes 4 fonctions phares (start, hello, recv_udp, shutdown) ont besoin de partager des informations entre elles :

start ouvre le socket UDP → il faut se souvenir du numéro du socket
hello veut envoyer un paquet → il a besoin du socket ouvert par start, et de l'UUID lu dans la config
recv_udp veut écouter les paquets → il a besoin du socket et de l'UUID aussi
shutdown doit fermer le socket → il a besoin du numéro du socket que start a ouvert

Question : comment ces fonctions se passent ces informations entre elles ?
Mauvaise solution : tout passer en paramètres
Tu pourrais écrire :
cint network_thread_hello(
    int udp_fd,
    uint8_t uuid[16],
    uint16_t udp_port,
    const char *broadcast_addr
);
Et pareil pour recv_udp, shutdown, etc. Mais ça devient vite ingérable :

Plein de paramètres à chaque appel
Si tu ajoutes une nouvelle info (par exemple le role), tu dois modifier toutes les signatures
Code lourd et bavard

Bonne solution : la structure net_state_t
On regroupe tout ce que doit savoir le Network Thread dans une seule structure. Regarde dans nt_types.h :
ctypedef struct {
    net_config_t    cfg;        /* la config (UUID, role, ports, ...) */
    int             udp_fd;     /* le socket UDP */
    int             epoll_fd;   /* l'epoll pour ecouter */
    service_qpair_t qpairs[10]; /* les queues POSIX */
    int             num_qpairs;
    volatile int    running;    /* 1 = tourne, 0 = arret demande */
} net_state_t;
Une seule structure qui contient tout l'état du Network Thread. C'est comme une "carte d'identité" qui voyage entre les fonctions.
Comment ça marche concrètement
1. On déclare UNE variable au début
Dans main.c :
cstatic net_state_t g_state;     /* la "carte d'identite" du Network Thread */
Cette variable existe une seule fois pour toute la durée du programme.
2. start la remplit
cnetwork_thread_start(&g_state, "worker.ini");
Le & signifie "donne-moi l'adresse de cette variable". On passe l'adresse à start pour qu'il puisse écrire dedans.
À l'intérieur de start :
cint network_thread_start(net_state_t *st, const char *config_path)
{
    config_load(config_path, &st->cfg);     /* remplit st->cfg avec le .ini */
    st->udp_fd = socket(AF_INET, SOCK_DGRAM, 0);  /* ecrit le socket dans st->udp_fd */
    st->running = 1;
    return 0;
}
Le *st veut dire "je manipule la variable dont on m'a donné l'adresse". Quand start se termine, g_state contient maintenant la config, le socket, etc.
3. hello lit dedans
cnetwork_thread_hello(&g_state);
On passe encore l'adresse de la même variable. Dans hello :
cint network_thread_hello(net_state_t *st)
{
    memcpy(hdr.src_uuid, st->cfg.uuid, 16);   /* lit l'UUID stocke par start */
    sendto(st->udp_fd, ...);                   /* utilise le socket stocke par start */
}
hello n'a pas besoin de redemander l'UUID ou le socket : il les trouve dans st (qui est en fait g_state).
4. shutdown ferme tout
cnetwork_thread_shutdown(&g_state);
cvoid network_thread_shutdown(net_state_t *st)
{
    close(st->udp_fd);     /* ferme le socket que start avait ouvert */
}
Analogie simple
Pense à net_state_t comme à un classeur partagé :

start ouvre le classeur et y range les documents (config, socket, queues)
hello consulte le classeur pour trouver l'UUID et le socket
recv_udp consulte le classeur pour trouver le socket d'écoute
shutdown vide le classeur et le ferme

Toutes les fonctions ont accès au même classeur, parce qu'on leur passe son adresse (&g_state).
Le * et le & en C
C'est LE point qui bloque tous les débutants. Voici la règle simple :

&variable veut dire "l'adresse de cette variable" (= "où est-elle rangée")
*pointeur veut dire "la valeur à cette adresse" (= "ce qui est rangé ici")
pointeur->champ veut dire "le champ de la structure à cette adresse"

Exemple :
cnet_state_t g_state;              /* variable qui existe en memoire */
net_state_t *st = &g_state;       /* st est un "pointeur" qui contient l'adresse de g_state */

st->udp_fd = 42;                  /* ecrit 42 dans g_state.udp_fd */
                                  /* equivalent a : g_state.udp_fd = 42 */
Pour répondre à ta question initiale

"mon start prend en entrée mon adresse ip"

Non. start prend en entrée deux choses :

net_state_t *st : l'adresse de la structure d'état (le "classeur")
const char *config_path : le nom du fichier de config (par exemple "worker.ini")

L'adresse IP n'est nulle part en paramètre. Elle est calculée automatiquement par le système quand le socket UDP bind() sur INADDR_ANY (toutes les interfaces réseau de la machine).

"je vois net_state partout"

C'est normal et c'est même voulu. Toutes les fonctions du Network Thread reçoivent ce net_state_t *st parce que c'est leur "contexte de travail commun". Sans ça, chaque fonction repartirait de zéro.
Le flux complet
cint main(int argc, char *argv[])
{
    net_state_t g_state;                    /* 1. on cree le classeur vide */

    network_thread_start(&g_state, argv[1]); /* 2. start le remplit */
    network_thread_hello(&g_state);          /* 3. hello l'utilise */

    while (g_state.running) {
        epoll_wait(...);
        network_thread_recv_udp(&g_state);   /* 4. recv_udp l'utilise */
    }

    network_thread_shutdown(&g_state);       /* 5. shutdown le vide */
    return 0;
}
Le même g_state voyage entre toutes les fonctions parce qu'on passe son adresse à chacune. C'est le pattern standard en C quand un module a un "état" à maintenir.

struct connection {
    struct sockaddr_in addr;    /* l'adresse complete (IP + port) deja prete pour sendto() */
    int                port;
    char               ip[16];
};

typedef struct {
    int               sockfd;         /* le socket UDP */
    struct connection self;           /* notre propre IP/port */
    struct connection broadcast;      /* la connexion de broadcast pour HELLO */
    uint8_t           uuid[16];
    volatile int      running;

c/* network_thread_start.c - version allégée */

struct connection create_connection(const char *ip, int port)
{
    struct connection c = {0};
    c.port = port;
    strncpy(c.ip, ip, 15);
    c.addr.sin_family = AF_INET;
    c.addr.sin_port = htons(port);
    inet_pton(AF_INET, ip, &c.addr.sin_addr);

AF_INET est une constante qui dit "c'est de l'IPv4". Sans ça, le système ne sait pas quel format d'adresse il manipule.
cc.addr.sin_port = htons((uint16_t)port);

cinet_pton(AF_INET, ip, &c.addr.sin_addr);

Ce qui change concrètement
1. Le rôle du Network Thread devient celui d'un pont
Sans queues (actuellement) :
Toi (humain) ──> ./parallax_netthread send "salut"  ──> reseau
Le binaire est un outil que tu lances à la main pour envoyer un truc.
Avec queues (avant et version PARALLAX réelle) :
Service local (execution, monitoring...) 
       │
       │ ecrit dans /parallax_execution_out (mq_send)
       ▼
[Network Thread] ──> reseau ──> [Network Thread distant]
                                         │
                                         │ ecrit dans /parallax_state_receiver_in
                                         ▼
                                  Service local (state_receiver)
Le binaire devient un service permanent qui écoute 2 sources en parallèle :

Le réseau (via recvfrom sur le socket UDP)
Les queues sortantes des services locaux (via mq_receive)

2. Tu auras besoin de plus de threads
Actuellement ton main.c a une boucle qui fait select() sur un seul descripteur : le socket UDP. Avec les queues, tu dois écouter en parallèle :

Le socket UDP
N queues sortantes (une par service local)

Deux solutions :
Solution A — Un thread sondeur dédié (ce qu'on avait avant) :
cpthread_t outq_thread;
pthread_create(&outq_thread, NULL, outq_thread_fn, &g_state);

// outq_thread_fn boucle sur toutes les queues _out avec mq_timedreceive
// et envoie sur le reseau via send_message()
Solution B — Boucle unique avec timeouts courts :
cwhile (running) {
    select(sockfd, timeout=50ms);  // reception reseau
    for each queue_out:
        mq_timedreceive(queue, timeout=10ms);  // lecture services
}
La solution A est plus propre mais ajoute un thread. La solution B est plus simple mais moins efficace.
